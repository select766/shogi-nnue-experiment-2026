# エキスパートの可視化

## 目的

Expert Blending で学習されたモデルにおいて、各エキスパートが異なる局面に
特化しているかどうかを可視化する。

## 入力と前提

- validation データ（学習スクリプトで使用したものと同じ）から先頭 10,000 局面程度を抽出する。
- 各局面 `i` に対して backbone を推論し、softmax 後のエキスパート重み `p(i, j)` を得る（`Σ_j p(i, j) = 1`）。
- 局面 `i` が対局中の何手目かを `m(i)` とする。
- 局面 `i` の教師評価値を `s(i)` とする。

## 出力

1 つのフォルダに HTML ファイル `index.html` と関連画像を出力する。
HTML はテーブル形式で、行ごとに 1 エキスパート、列は次の 4 つ。

| 列 | 内容 |
|---|---|
| Expert | エキスパート番号と担当局面数 |
| Ply histogram | 担当局面の手数分布ヒストグラム（後述） |
| Score histogram | 担当局面の教師評価値分布ヒストグラム（後述） |
| Representative positions | 代表局面 10 個の盤面画像（後述） |

### 担当局面と手数分布ヒストグラム

エキスパート `j` に「割り当てられた」局面とは
`j = argmax_{j'} p(i, j')` を満たす `i` のこと。
担当局面の手数 `m(i)` を横軸、局面数を縦軸にしてヒストグラムを描く。

### 評価値分布ヒストグラム

担当局面の教師評価値 `s(i)` を横軸、局面数を縦軸にしてヒストグラムを描く。

### 代表局面

`p(:, j)` の上位だけを取ると、gate が極端に強く反応した
adversarial な局面に偏り、ヒストグラムが示す「担当局面の典型像」と
食い違う。これを避けるため、次の手順で 10 局面を選ぶ。

1. **担当局面に絞る**: `argmax_{j'} p(i, j') = j` を満たす `i` のみを対象とする。
2. **中央値近傍を取る**: 残った局面を `p(i, j)` の昇順で並べ、中央の 10 個を選ぶ。

こうすると、エキスパートが「ふつうに選んでいる」局面が代表として抽出される。
局面は SFEN に変換した上で画像（SVG）にレンダリングする。

## 将棋特徴量 Lift 分析

### 目的

各エキスパートが「どのような局面を担当しているか」を将棋用語で説明する。
統計的根拠として **lift**（リフト）を使い、局面特徴との関連度を定量化する。

### Lift の定義

エキスパート `j` と特徴 `f` の lift を次のように定義する。

```
lift(f, j) = P(f | dominant=j) / P(f)
```

- `P(f)` : 全局面における特徴 `f` の出現率
- `P(f | dominant=j)` : エキスパート `j` が担当する局面（`argmax p(i,j') = j`）における `f` の出現率
- lift = 1.0 → 全体と同じ割合（関連なし）
- lift = 3.0 → 全体の 3 倍の頻度で担当（正の関連）
- lift = 0.3 → 全体の 1/3 しか担当しない（負の関連）

Wilson score 95% CI を同時に算出し、**CI 下限 ≥ 1.5** のときのみ
「有意な正の関連あり」と判定してエキスパートの自動ラベルに採用する。

### 局面特徴の定義

すべての特徴は bool（True / False）値として各局面に付与する。

#### 戦型（飛車の筋）

cshogi の `board.piece(sq)` で飛車（`BROOK`/`WROOK`）または竜（`BPROM_ROOK`/`WPROM_ROOK`）
を検索し、`cshogi.make_file(sq) + 1` で将棋の筋番号（1–9）を得る。

| 特徴名 | 定義 | 備考 |
|---|---|---|
| `b_ibisha` | 先手飛車が 1–4 筋 | 居飛車（初期位置は 2 筋） |
| `b_chubi` | 先手飛車が 5 筋 | 中飛車 |
| `b_furibisha` | 先手飛車が 6–9 筋 | 振り飛車（四間・三間・向かい飛車を含む） |
| `w_ibisha` | 後手飛車が 6–9 筋 | 後手居飛車（初期位置は 8 筋） |
| `w_chubi` | 後手飛車が 5 筋 | 後手中飛車 |
| `w_furibisha` | 後手飛車が 1–4 筋 | 後手振り飛車 |
| `ai_furibisha` | `b_furibisha` かつ `w_furibisha` | 相振り飛車 |
| `taiko_form` | (`b_furi` かつ `w_ibisha`) または (`b_ibisha` かつ `w_furi`) | 対抗形 |

- 飛車が盤上にない（相手に取られた、または竜に成って位置が変わった場合も含む）
  局面では、いずれの戦型特徴も False となる。
- 竜（成り飛車）はその位置で飛車と同様に戦型判定に使う。

#### 入玉

`cshogi.make_rank(sq)` は 0-indexed で 0 = 1 段目（後手陣）、8 = 9 段目（先手陣）。

| 特徴名 | 定義 |
|---|---|
| `b_irikoma` | 先手玉の rank ≤ 2（1–3 段目 = 後手陣に入った） |
| `w_irikoma` | 後手玉の rank ≥ 6（7–9 段目 = 先手陣に入った） |
| `irikoma_any` | `b_irikoma` または `w_irikoma` |

#### 穴熊ヒント

囲いを厳密にパターンマッチするのではなく、玉位置の簡易ヒューリスティック。

| 特徴名 | 定義 |
|---|---|
| `b_anaguma_hint` | 先手玉が端筋（1–2 筋 or 8–9 筋）かつ rank ≥ 7（8–9 段目） |
| `w_anaguma_hint` | 後手玉が端筋（1–2 筋 or 8–9 筋）かつ rank ≤ 1（1–2 段目） |

穴熊以外の端玉（例：端攻め後に玉が移動した局面）も True になりうる点に注意。

#### 手数帯

局面に付随する手数 `m(i)`（`ply`）をそのまま利用する。

| 特徴名 | 定義 |
|---|---|
| `early_game` | ply ≤ 40 |
| `mid_game` | 40 < ply ≤ 100 |
| `late_game` | ply > 100 |

#### 評価値帯

教師評価値 `s(i)`（`score`）の絶対値で判定する。

| 特徴名 | 定義 |
|---|---|
| `score_even` | \|s\| ≤ 500（互角） |
| `score_winning` | \|s\| > 2000（勝勢） |

### 自動ラベルの生成規則

1. 全特徴を `|log(lift)|` の降順でソートする。
2. 優先順位リスト（`_LABEL_PRIORITY`）の順に走査し、
   **CI 下限 ≥ 1.5** を満たす特徴を上位 2 つ選ぶ。
3. ラベル文字列は `"特徴名(lift値×)"` を `/` で連結する（例: `入玉あり(3.4×) / 終盤(2.3×)`）。
4. 有意な特徴が 1 つもなければ `—` を表示する。

優先順位は `irikoma_any` > `ai_furibisha` > `taiko_form` > 振り飛車系 >
穴熊系 > 手数帯 > 評価値帯 の順（`_LABEL_PRIORITY` リストで管理）。

### 出力ファイル

| ファイル | 内容 |
|---|---|
| `index.html` | エキスパート一覧 HTML（lift ラベル・折り畳み詳細を含む） |
| `lift_analysis.json` | 全エキスパートの lift テーブル・統計を含む機械可読 JSON |
| `feature_check.html` | 各特徴の True/False 局面サンプルを各 10 個表示する検証 HTML |
| `feat_check_{n}.svg` | `feature_check.html` 用の局面 SVG 画像 |

- 可視化対象のモデル・実行コマンド・出力先などは
